Understanding Rotation Mathematics

In embedded systems like drone flight controllers or robotics, representing 3D orientation accurately and efficiently is critical. We primarily use two methods: Euler Angles and Quaternions.

1. Euler Angles

Euler angles represent rotation as a sequence of three rotations around the axes of a coordinate system. A common convention is the ZYX sequence (Yaw, Pitch, Roll):

\[ R = R_z(\psi) R_y(\theta) R_x(\phi) \]

The Pros

  • Intuitive: Easy for humans to visualize (e.g., "turn 90 degrees left").
  • Compact: Only 3 numbers required.

The Cons: Gimbal Lock

When the pitch angle \(\theta\) reaches \(\pm 90^\circ\), the first and third axes align. This causes a "loss of a degree of freedom," where changing roll and yaw results in the same physical motion. Mathematically, the mapping from angular rates to orientation becomes singular.

2. Quaternions

A quaternion is a four-dimensional complex number used to represent rotations in 3D space. It is defined as:

\[ \mathbf{q} = w + xi + yj + zk \] \[ \text{where } i^2 = j^2 = k^2 = ijk = -1 \]

For a rotation of angle \(\theta\) around a unit axis \(\vec{u} = (u_x, u_y, u_z)\), the unit quaternion is:

\[ \mathbf{q} = \left( \cos\frac{\theta}{2}, u_x\sin\frac{\theta}{2}, u_y\sin\frac{\theta}{2}, u_z\sin\frac{\theta}{2} \right) \]

The Rotation Formula

To rotate a vector \(\vec{v}\), we treat it as a pure quaternion \(\mathbf{v} = (0, v_x, v_y, v_z)\) and calculate the sandwich product:

\[ \mathbf{v}' = \mathbf{q} \mathbf{v} \mathbf{q}^{-1} \]

3. Implementation in C

The library is designed for spatial locality and computational efficiency, ensuring that the hardware can process vectors and quaternions at maximum speed.

A. Memory Layout & Data Structures

We use simple structures to ensure that components are stored contiguously in memory. This allows the compiler to use "Load Multiple" (LDM) and "Store Multiple" (STM) instructions, which are highly efficient on ARM architectures.

typedef struct { float x; float y; float z; } vec3_t;
typedef struct { float w; float x; float y; float z; } quat_t;

B. Quaternion Multiplication (Hamilton Product)

Combining two rotations is equivalent to multiplying their quaternions. This is implemented as a 4D complex product:

w = q1.w*q2.w - q1.x*q2.x - q1.y*q2.y - q1.z*q2.z;
x = q1.w*q2.x + q1.x*q2.w + q1.y*q2.z - q1.z*q2.y;
y = q1.w*q2.y - q1.x*q2.z + q1.y*q2.w + q1.z*q2.x;
z = q1.w*q2.z + q1.x*q2.y - q1.y*q2.x + q1.z*q2.w;

4. Real-Time Optimizations for 32-bit MCUs

To run these calculations real-time on an STM32 (Cortex-M4/M7), several optimizations were implemented in this library:

A. The Optimized Sandwich Product

The standard \(\mathbf{q}\mathbf{v}\mathbf{q}^{-1}\) requires 24 multiplications and 17 additions. We use an optimized vector-based formula:

\[ \vec{v}' = \vec{v} + 2\vec{q}_{xyz} \times (\vec{q}_{xyz} \times \vec{v} + w\vec{v}) \]

This reduces the computational load significantly by utilizing the cross product, which is very efficient on 32-bit registers.

B. Avoiding Double-Precision Promotion

Most STM32 chips (Cortex-M4F) have a Single-Precision FPU. In C, 2.0 is a double. Using it forces the MCU to simulate double-precision in software, which is 10-100x slower. We strictly use:

  • float literals: 2.0f
  • Single-precision functions: sqrtf(), sinf(), cosf()

C. Q30 Fixed-Point Arithmetic

For chips without an FPU (Cortex-M0/M3) or for maximum determinism, we implemented Q30 Fixed-Point. This achieves floating-point results using only integer operations by scaling the continuous real-number space.

1. The Scaling Factor

In Q30 format, the value \(1.0\) is represented as \(2^{30}\) (\(1,073,741,824\)). This provides a resolution of approximately \(9.3 \times 10^{-10}\), which is actually higher than the precision of a standard 32-bit float.

2. Mathematical Emulation

Multiplying two scaled integers doubles the scaling factor (\(2^{30} \times 2^{30} = 2^{60}\)). To return to Q30 space, we must shift the result:

static inline q30_t q30_mul(q30_t a, q30_t b) {
    return (q30_t)(((int64_t)a * b) >> 30);
}

By using a 64-bit intermediate (int64_t), we ensure no data is lost during the multiplication before the shift.

3. Comparison of Accuracy

Since quaternions and normalized vectors are bounded between -1 and 1, the Q30 format provides superior precision for this specific application while significantly increasing execution speed.

Conclusion: While Euler angles are better for UI display, Quaternions are the mathematically superior choice for internal estimation and control loops, especially when optimized for the underlying hardware architecture.

5. Sensor-to-Body Transformation & Calibration

In real-world applications, the IMU is rarely aligned perfectly with the vehicle's body axes. We must transform the sensor's local readings \(\vec{v}_{sensor}\) into the vehicle's body frame \(\vec{v}_{body}\).

A. Basis Vector Extraction

The core of the calibration logic is determining the orientation of the sensor frame's unit axes relative to the body frame. We rotate the three basis vectors \(\hat{i}, \hat{j}, \hat{k}\) of the sensor frame using the current orientation quaternion \(\mathbf{q}\):

\[ \vec{c}_0 = \text{Rotate}(\mathbf{q}, [1, 0, 0]^T) \] \[ \vec{c}_1 = \text{Rotate}(\mathbf{q}, [0, 1, 0]^T) \] \[ \vec{c}_2 = \text{Rotate}(\mathbf{q}, [0, 0, 1]^T) \]

These vectors \(\vec{c}_0, \vec{c}_1, \vec{c}_2\) form the columns of the 3x3 rotation matrix \(R_{final}\). Each column tells you where a specific sensor axis points in the body frame.

B. Automatic Rotation Decomposition

To make calibration easier to manage, the tool automatically decomposes the final mounting matrix \(R_{final}\) into two distinct components:

\[ R_{final} = R_{rough} \times R_{precise} \]

1. Rough Calibration (\(R_{rough}\))

This represents the **cardinal alignment** (the 90-degree axis swaps and flips). It is found by identifying the dominant body axis for each sensor basis vector:

  • Find the component of \(\vec{c}_0\) with the maximum absolute value. Set that as the primary axis of \(\vec{r}_0\) with a magnitude of \(\pm 1\).
  • Repeat for \(\vec{c}_1\) (ensuring a different axis is chosen).
  • Calculate \(\vec{r}_2 = \vec{r}_0 \times \vec{r}_1\) to ensure a right-handed system.

2. Precise Calibration (\(R_{precise}\))

This represents the **residual error** or fine-tuning (e.g., a 1.5-degree mounting tilt). Since \(R_{rough}\) is an orthogonal matrix, its inverse is its transpose. We calculate the residual as:

\[ R_{precise} = R_{rough}^T \times R_{final} \]

This separation allows engineers to distinguish between "intended mounting" (Rough) and "mounting tolerances" (Precise).

C. Optimized Vector Rotation Formula

To perform these rotations efficiently on embedded hardware, the library avoids the standard quaternion multiplication \(\mathbf{q}\mathbf{v}\mathbf{q}^{-1}\) (24 multiplications) and uses a vector-optimized identity:

\[ \vec{v}' = \vec{v} + 2\vec{q}_{xyz} \times (\vec{q}_{xyz} \times \vec{v} + w\vec{v}) \]

This formula requires only 15 multiplications and 12 additions, providing a significant performance boost in the IMU data processing pipeline.

6. From User Input to Quaternions

When a user inputs Euler angles (Roll, Pitch, Yaw), the library converts them into a unit quaternion \(\mathbf{q}\). It is important to note that the sensor's position \((x, y, z)\) does not affect the orientation math; it only shifts the origin of the transformation.

A. The ZYX Convention

We use the **ZYX intrinsic rotation convention** (also known as the 3-2-1 sequence), which is the standard in aerospace and navigation. The rotation is performed in the following order:

  1. Yaw (\(\psi\)): Rotation around the Z-axis.
  2. Pitch (\(\theta\)): Rotation around the new Y-axis.
  3. Roll (\(\phi\)): Rotation around the new X-axis.

B. Conversion Formula

Given the angles in radians, the components of the quaternion \(\mathbf{q} = [w, x, y, z]\) are calculated as follows:

\[ w = \cos\frac{\phi}{2}\cos\frac{\theta}{2}\cos\frac{\psi}{2} + \sin\frac{\phi}{2}\sin\frac{\theta}{2}\sin\frac{\psi}{2} \] \[ x = \sin\frac{\phi}{2}\cos\frac{\theta}{2}\cos\frac{\psi}{2} - \cos\frac{\phi}{2}\sin\frac{\theta}{2}\sin\frac{\psi}{2} \] \[ y = \cos\frac{\phi}{2}\sin\frac{\theta}{2}\cos\frac{\psi}{2} + \sin\frac{\phi}{2}\cos\frac{\theta}{2}\sin\frac{\psi}{2} \] \[ z = \cos\frac{\phi}{2}\cos\frac{\theta}{2}\sin\frac{\psi}{2} - \sin\frac{\phi}{2}\sin\frac{\theta}{2}\cos\frac{\psi}{2} \]

Every resulting quaternion is normalized such that \(w^2 + x^2 + y^2 + z^2 = 1\), ensuring it represents a valid rotation in 3D space.

Conclusion: While Euler angles are better for UI display, Quaternions are the mathematically superior choice for internal estimation and control loops, especially when optimized for the underlying hardware architecture.
